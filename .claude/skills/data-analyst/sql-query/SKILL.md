## SQL Best Practices Applied

**Readability:**
- CTEs over nested subqueries (easier to debug and maintain)
- Consistent indentation and capitalization (keywords UPPERCASE)
- Meaningful aliases (not `t1`, `t2` — use `orders`, `customers`)
- Comments explaining WHY, not WHAT (the SQL shows what)

**Performance:**
- Filter early (WHERE in CTEs, not final SELECT)
- Avoid SELECT * — specify columns explicitly
- Use EXISTS instead of IN for large subqueries
- Prefer JOIN over correlated subqueries
- Use appropriate date functions for the dialect
- LIMIT for exploration queries

**Common Patterns:**
| Pattern | When to Use |
|---------|-------------|
| Window Functions (ROW_NUMBER, RANK, LAG, LEAD, SUM OVER) | Rankings, running totals, period-over-period |
| CTEs (WITH clause) | Multi-step logic, readability |
| CASE WHEN | Conditional aggregation, bucketing, pivoting |
| DATE_TRUNC / DATE_DIFF | Time-based grouping and comparison |
| LEFT JOIN ... WHERE IS NULL | Anti-join (finding records that DON'T exist) |
| COALESCE / NULLIF | Null handling |
| HAVING | Filtering aggregated results |

## Dialect Adaptations

| Feature | PostgreSQL | BigQuery | Snowflake | MySQL |
|---------|-----------|----------|-----------|-------|
| Date truncation | DATE_TRUNC('month', d) | DATE_TRUNC(d, MONTH) | DATE_TRUNC('MONTH', d) | DATE_FORMAT(d, '%Y-%m-01') |
| String concat | `\|\|` | CONCAT() | `\|\|` | CONCAT() |
| LIMIT | LIMIT N | LIMIT N | LIMIT N | LIMIT N |
| Array agg | ARRAY_AGG() | ARRAY_AGG() | ARRAY_AGG() | GROUP_CONCAT() |
| Regex | ~ | REGEXP_CONTAINS | RLIKE | REGEXP |

## Rules

- Always specify the SQL dialect (default PostgreSQL if not stated)
- Use CTEs for multi-step queries — never nest more than 2 levels
- Comment complex logic but don't over-comment obvious operations
- Include expected output shape (sample row)
- Provide performance notes for complex queries
- Suggest indexes when the query would benefit from them
- Handle NULLs explicitly — never assume clean data
- Offer variations for common follow-up questions
- If table structure is unknown, clearly state assumptions and use generic names
